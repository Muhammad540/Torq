Limits

Limits define inequality constraints in the QP. They ensure the solved \(\Delta q\) respects the physical bounds of the robot. Each limit produces rows \((G, h)\) such that \(G\,\Delta q \le h\). See QP Formulation for how constraints compose and Conventions and Notation for notation.

Limits are hard constraints — the QP solver will never return a \(\Delta q\) that violates them. If no feasible solution exists, the solver reports infeasibility.

Torq provides 4 limit types. Two are built into every RobotSystem (VelocityLimit, ConfigurationLimit); the others are added via RobotSystem::addLimit().

Limit overview

Limit Rows Purpose Built-in?
VelocityLimit \(2k\) Joint velocity bounds Yes
ConfigurationLimit \(2k\) Joint position bounds with smooth deceleration Yes
FloatingBaseVelocityLimit \(2 \times 6\) Base twist limits for mobile/legged robots No
AccelerationLimit \(2k\) Joint acceleration bounds with braking distance No

Common interface

All limits inherit from torq::Limit and implement a single method:

virtual std::optional<std::pair<Eigen::MatrixXd, Eigen::VectorXd>>
computeQPInequalities(const Configuration& config, double dt) const = 0;

Returns std::nullopt if there are no active constraints (e.g. no finite bounds in the model), in which case the limit contributes zero rows.

VelocityLimit

Purpose: Bound joint displacements so that the implied velocity does not exceed hardware limits.

Formulation

Given the maximum velocity vector \(\dot{q}_{\max} \in \mathbb{R}^{n_v}\) (from model.velocityLimit) and the integration timestep \(\Delta t\):

\[ -\Delta t\;\dot{q}_{\max} \;\le\; \Delta q \;\le\; \Delta t\;\dot{q}_{\max} \]

Written in standard inequality form \(G\,\Delta q \le h\) this becomes \(2k\) rows (upper + lower bound for each bounded DOF):

\[ \begin{bmatrix} +S \\ -S \end{bmatrix} \Delta q \;\le\; \begin{bmatrix} \Delta t\;\dot{q}_{\max}^{(k)} \\ \Delta t\;\dot{q}_{\max}^{(k)} \end{bmatrix} \]

where \(S \in \mathbb{R}^{k \times n_v}\) is a selection matrix that picks out the \(k\) joints with finite velocity bounds. Joints without a finite velocityLimit in the URDF/MJCF are excluded.

How it works in code

  1. On construction, VelocityLimit scans the Pinocchio model for joints with finite velocity bounds and builds a projection matrix \(S\).
  2. Each tick, computeQPInequalities() scales the bounds by \(\Delta t\) and returns the stacked \((G, h)\).

Parameters

None — limits are read directly from the Pinocchio model. To change velocity limits, modify the URDF or the model before constructing the limit.

Key property

Velocity limits scale correctly with the timestep \(\Delta t\). A larger timestep allows larger per-step displacements. The actual velocity bound \(\dot{q}_{\max}\) is independent of the control frequency.

ConfigurationLimit

Purpose: Keep \(q + \Delta q\) within the joint position bounds \([q_{\min}, q_{\max}]\), using a proportional gain to steer away from limits before they are hit.

Formulation

For each bounded joint \(j\) with current position \(q_j\) and limits \(q_j^{\min}, q_j^{\max}\):

\[ \Delta q_j \;\le\; \gamma\,(q_j^{\max} \ominus q_j) \]

\[ -\Delta q_j \;\le\; \gamma\,(q_j \ominus q_j^{\min}) \]

where \(\gamma \in (0, 1]\) is the configuration limit gain (config_limit_gain) and \(\ominus\) is the manifold-aware difference. For revolute joints, \(q_j^{\max} \ominus q_j = q_j^{\max} - q_j\).

In matrix form for all \(k\) bounded joints:

\[ \begin{bmatrix} +S \\ -S \end{bmatrix} \Delta q \;\le\; \begin{bmatrix} \gamma\,(q^{\max}_{(k)} \ominus q_{(k)}) \\ \gamma\,(q_{(k)} \ominus q^{\min}_{(k)}) \end{bmatrix} \]

How the gain works

The gain \(\gamma\) controls how much of the remaining range the solver is allowed to use in a single tick:

\(\gamma\) Behaviour
1.0 Allow the full remaining range — only prevents actual violation.
0.5 (default) Permit half the remaining range per tick — smooth deceleration near limits.
0.1 Very conservative — robot slows down far from limits.

Lower gain → earlier and smoother deceleration as the robot approaches a limit. This prevents hard stops and reduces stress on hardware.

How it works in code

  1. On construction, scans for joints with finite position bounds.
  2. Each tick, computes pinocchio::difference(model, q, q_max) and pinocchio::difference(model, q, q_min) to get manifold-aware slack.
  3. Scales the slack by \(\gamma\) and returns the stacked constraints.

Parameters

Parameter Symbol Default Setter
config_limit_gain \(\gamma\) 0.5 setConfigLimitGain(double)

FloatingBaseVelocityLimit

Purpose: Cap the linear and angular velocity of the floating base for mobile or legged robots.

Formulation

Given maximum twist \(v_{\max} \in \mathbb{R}^6\) (linear + angular):

\[ -\Delta t\,v_{\max} \;\le\; J_{\text{root}}\,\Delta q \;\le\; \Delta t\,v_{\max} \]

In standard form:

\[ \begin{bmatrix} +J_{\text{root}} \\ -J_{\text{root}} \end{bmatrix} \Delta q \;\le\; \begin{bmatrix} \Delta t\,v_{\max} \\ \Delta t\,v_{\max} \end{bmatrix} \]

The Jacobian \(J_{\text{root}}\) is the frame Jacobian of the base with all columns outside the root joint zeroed. This ensures only the floating-base DOFs are constrained.

Requirements

  • The model must have a root_joint (floating base).
  • A base frame name can be provided, or it is auto-detected.

How it works in code

  1. On construction, identifies the root joint and its velocity indices.
  2. Each tick, computes the frame Jacobian, zeros non-root columns, and builds the box constraint.
  3. Only rows with finite bounds are included.

Parameters

Parameter Type Description
base_frame string Base frame name (empty = auto-detect).
max_linear_velocity Vector3d or double Linear velocity bound [m/s].
max_angular_velocity Vector3d or double Angular velocity bound [rad/s].

Example

auto fb_limit = std::make_unique<torq::FloatingBaseVelocityLimit>(
robot.model(),
"", // auto-detect base frame
0.5, // max linear velocity [m/s]
1.0 // max angular velocity [rad/s]
);
robot.addLimit(std::move(fb_limit));

AccelerationLimit

Purpose: Limit joint accelerations to smooth motion and prevent torque spikes. Combines a finite-difference acceleration bound with a braking-distance constraint.

Formulation

Two constraint types are enforced per joint, and the tighter one wins:

1. Finite-difference acceleration bound:

\[ \Delta q_{\text{prev}} - a_{\max}\,\Delta t^2 \;\le\; \Delta q \;\le\; \Delta q_{\text{prev}} + a_{\max}\,\Delta t^2 \]

This limits how much the displacement can change from the previous tick, bounding the finite-difference acceleration.

2. Braking-distance bound (Flacco 2015, Del Prete 2018):

\[ |\dot{q}| \;\le\; \sqrt{2\,a_{\max}\,d_{\text{limit}}} \]

where \(d_{\text{limit}}\) is the distance to the nearest joint position limit. This ensures the robot can always decelerate to a stop before hitting the limit, given the maximum deceleration \(a_{\max}\).

In displacement terms:

\[ \Delta q_{\max}^{\text{brake}} = \Delta t \cdot \sqrt{2\,a_{\max}\,\max(d_{\text{limit}}, 0)} \]

The effective bound is:

\[ h = \min\left(a_{\max}\,\Delta t^2 + \Delta q_{\text{prev}},\; \Delta t\,\sqrt{2\,a_{\max}\,d_{\text{limit}}}\right) \]

Requirements

setLastIntegration(v_prev, dt) must be called each tick with the velocity from the previous IK solve. When added via RobotSystem::addLimit(), the controller handles this automatically.

Parameters

Parameter Type Description
acceleration_limit VectorXd Per-joint max acceleration (dimension \(n_v\)).

Example

Eigen::VectorXd a_max = Eigen::VectorXd::Constant(robot.nv(), 10.0);
auto accel_limit = std::make_unique<torq::AccelerationLimit>(
robot.model(), a_max);
robot.addLimit(std::move(accel_limit));

How limits stack

All limit types produce rows in the same inequality system:

\[ G = \begin{bmatrix} G_{\text{vel}} \\ G_{\text{cfg}} \\ G_{\text{fb}} \\ G_{\text{accel}} \end{bmatrix}, \quad h = \begin{bmatrix} h_{\text{vel}} \\ h_{\text{cfg}} \\ h_{\text{fb}} \\ h_{\text{accel}} \end{bmatrix} \]

A limit may return std::nullopt if it has no active constraints, in which case it contributes zero rows. Barrier constraint rows are also stacked into the same system (see Barriers).

The OSQP solver handles the full stacked system simultaneously.

Planned: equality constraints

Some tasks may in the future act as hard equality constraints rather than soft objectives:

\[ A_{\text{eq}}\,\Delta q = b_{\text{eq}} \]

The QPProblem struct is designed to be extended with equality rows.
